## 2025-05-07 - Refactoring Promise.all in loops

**Learning:** When multiple arrays of independent async I/O promises (like file reads for configs at different hierarchy levels) are awaited sequentially (`await Promise.all(group1)`, then `await Promise.all(group2)`), it introduces a waterfall. However, a naive loop refactor `for (const x of await Promise.all(promises))` vs `const items = await Promise.all(promises); for (const x of items)` provides zero performance benefit and is functionally identical. The actual optimization requires hoisting all the promise groups and awaiting them together in an outer `Promise.all([Promise.all(group1), group2, Promise.all(group3)])` before iterating, which allows all config tiers to be fetched concurrently over the network or file system.

**Action:** When parallelizing independent operations that must be processed in a specific sequential order, instantiate all the promises first, use a single outer `Promise.all()` to fetch them concurrently, and then sequentially iterate over the resolved structures. Avoid naive syntactical refactors of `for...of` loops that don't change execution concurrency.
